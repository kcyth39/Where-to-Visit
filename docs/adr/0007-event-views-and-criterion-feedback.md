# ADR-0007: イベント画面の分離と判断基準別フィードバック

- **ステータス:** Accepted
- **日付:** 2026-07-12
- **決定者:** おしげさん
- **関連:** [ADR-0003](0003-evaluation-and-decision-logic.md) / [ADR-0004](0004-permission-model.md) / [ADR-0005](0005-drop-attribute-dynamic-criteria.md) / [ADR-0006](0006-collaborative-response-row-model.md) / [03_requirements](../03_requirements.md) / [04_data-model](../04_data-model.md)
- **実装状態（2026-07-13）:** ADR-0006移行と同じ実装でコード・DB・UIへ反映済み。候補一覧と候補編集route、Criterion別Concernを検証済み

## コンテキスト

ADR-0006では、Event詳細に回答者セレクター、判断基準、全Candidateカード、全回答者行をまとめる構造を採用していた。しかし、共有URLを開いた人が最初に行う名前選択、候補全体を眺める操作、1候補を詳しく吟味・編集する操作は目的が異なる。同じ画面へすべてを載せると、説明と操作が増え、○ / − / ×と補助情報の優先順位も伝わりにくい。

また、従来の🌀はCandidate単位の常設単一フラグだったが、実際には「価格は気になる」「雰囲気は好き」のように、同じ判断基準に対して❤️と🌀の両方を独立して表現する必要がある。Candidate単位の🌀では、何に懸念があるのかを回答者間で共有できない。

## 決定

### 1. 画面と用語を分離する

- **トップ**: きめることと任意のつたえておきたいことを作成する画面。イベント内の「候補一覧」リンクは表示しない。
- **イベント一覧**: 同じブラウザに保存された複数Eventをトップ下部へ並べ、再訪するための将来機能。本スライスでは実装しないが、トップのレイアウトは将来追加できる構造にする。
- **候補一覧ダッシュボード**: 1つのEventに属するCandidateを見渡し、みんなの意見を確認する画面。共有URLの通常の閲覧先とする。
- **候補編集画面**: 1つのCandidateを詳しく吟味し、判断基準、回答者別の○ / − / ×、❤️、🌀、コメント、Candidate情報を共同編集する画面。Candidate名から開く。

「トップ / 未選択 / 選択済み / 確認画面」を上部タブとして並べない。ブランド「きめのすけ」はトップへの導線とする。Event文脈のある画面では候補一覧ダッシュボードへ戻る導線を置くが、トップには置かない。

`/e/<share_token>`と`/o/<owner_token>`は外部共有・owner回復の入口として維持する。本ADRはユーザーが見る論理画面と遷移を定義し、Candidate編集を内部routeまたは同一route内viewのどちらで実現するかは、共有URL契約を変えない範囲の実装詳細とする。

### 2. Event作成後と共有URLの入口を分ける

- トップのEvent作成フォームは「きめること → つたえておきたいこと（任意） → きめよう！」だけとし、名前と属性を置かない。
- オーナーはEvent作成後、同じEventの初期セットアップ画面へ進む。Event情報の下にお名前と候補の初期入力を表示し、入力完了後の「さあ、きめよう！」で同じ画面の初回共有ステップへ進む。みんなに送るリンクを先に表示し、「わたしの意見を入力」で同じタブの候補一覧ダッシュボードへ進む。URL共有は候補一覧ダッシュボードにも置く。
  1. **お名前を入れる**: 「ここで選んだ名前が、候補や回答の名義になります。」
  2. **候補の追加**: 「候補名だけでも、リンクだけでも追加できます。」
- オーナーが名前を確定しても画面を離れず、Candidate追加へ続ける。追加済みCandidateを画面内へ表示し、名前とCandidateがそろうまで開始ボタンを無効にする。
- 共有URLを開き、選択中Participantがないゲストには名前選択だけを表示する。既存名の選択肢の直下に「直接入力」を置き、既存名を押した場合は入力欄にも反映する。
- ゲストが既存名を選択または新名を確定した後は、候補一覧ダッシュボードへ進む。
- localStorageに同一Eventの現存Participant IDがあれば、共有URLから候補一覧ダッシュボードへ直接進む。無効・削除済みIDなら選択を解除して名前選択を表示する。
- Event作成直後の初期セットアップはその作成セッションだけに表示する。別ブラウザまたは後日のowner URL検証成功時は、selected participantがなくても候補一覧ダッシュボードを表示し、きめること・つたえておきたいことの編集を可能にする。
- 初期セットアップ完了状態はDBへ保存しない。作成成功直後の一時的な画面modeとし、reload・再訪では候補一覧へ進む。
- owner URLで再訪した未選択オーナーが○ / − / ×、❤️ / 🌀、コメント等の個人名義操作を始めた場合だけ、操作を保留して名前選択へ進み、Participant解決後に一度だけ再開する。

### 3. 候補一覧ダッシュボードはサマリーへ集約する

- 画面上部にEventのきめることとつたえておきたいことを表示する。スクロール後に固定表示する必要はない。
- Candidateは1候補1行のサマリー表へ集約し、Candidate名、URL、総合評価の`⭕️ / ➖ / ❌`と各件数、❤️ / 🌀合計を表示する。Candidate名は候補編集への導線とする。
- `➖`件数は能動的な`neutral` Vote行だけを数え、Vote行なしの`unrated`を含めない。未評価は候補編集の回答者行で別表示する。
- 追加時期と提案者は候補編集画面で確認し、ダッシュボードへ重複表示しない。
- ○ / − / ×は決定に関与する主要情報、❤️ / 🌀は参考情報として視覚的な優先順位を分ける。
- `clear / discussion / fallback / none`の説明ラベル（例:「○最多・×なし」）は画面へ表示しない。サマリー行の控えめな背景・境界色で状態を示し、支援技術向けにはsemantic stateを読める名前を付ける。○ / − / ×の実数は常に表示する。
- サマリー表の下へ同内容のCandidateカードを重ねない。将来の広告領域は本スライスでは実装しない。

### 4. 候補編集画面は1候補を吟味する

- Candidate名、URL、追加時期、提案者を表示し、その直下でサマリーと同じ○ / − / ×、❤️ / 🌀を選択中回答者名義で操作する。
- 選択中回答者は候補タイル直前に「〇〇として判断中」と表示する。コメント入力は候補タイル内の評価control直下へ置き、明示保存する。
- 全回答者行は「みんなの判断」にread-onlyで並べ、総合評価、判断基準別❤️ / 🌀、コメント全文を表示する。行clickによる回答者変更や行内編集controlは置かない。
- 「みんなの判断」の下へ「候補内容の編集」「❤️／🌀反応項目の編集」「判断者名の変更／削除」を置く。候補内容だけを＋／−付きの開閉UIからインラインpanelで開き、反応項目編集と判断者名の変更／削除はmodalで開く。modal導線2件はデスクトップで同一行に並べて文言を改行せず、モバイルで横幅不足時だけボタン単位・文言とも折り返せる。反応項目編集は既存項目一覧の下に追加buttonを置き、プリセット・自由入力の追加、名前変更、2段階削除を含む。判断者名の変更／削除は選択中回答者だけを扱い、現在名を直接編集できるinputと、変更・キャンセル・右端の削除を同一modalへ置く。削除確認中は編集UIを隠し、「消す／キャンセル」だけを表示する。
- 候補削除は候補内容の編集、回答者削除は判断者名の変更／削除へ収め、画面上へ独立した削除buttonを置かない。サマリーの反応入力からも共通の反応項目編集modalへ進める。
- 変更・同名確認・削除確認は、その時点の対象について1画面ずつ表示する。複数の確認画面をタイル状に並べない。
- モバイルは回答者行を縦方向へまとめるが、コメントを省略せず全文表示する。デスクトップと操作モデルは変えない。

### 5. 🌀を判断基準単位へ置き換える

- Candidate単位で独立していた常設単一Concernを廃止する。
- ❤️と🌀は、どちらもCandidate×Participant×Criterionごとの独立した2値とする。
- 同じParticipantが同じCandidate・Criterionへ❤️と🌀を両方付けられる。
- ❤️は`reactions`、🌀は`concerns`として別テーブルを維持し、いずれも`UNIQUE(candidate_id, participant_id, criterion_id)`とする。
- Concernへ`criterion_id NOT NULL REFERENCES criteria(id) ON DELETE CASCADE`を追加し、Candidate / Participant / Criterionが同じEventに属することをDBで保証する。
- Candidate全体の❤️数はReaction行数、🌀数はConcern行数の単純合計とする。判断基準ごとの件数と付与者も表示する。
- ❤️と🌀は最終候補判定へ影響しない。UPDATEや履歴を持たず、行のINSERT / DELETEで現在値を表す。

## 置換範囲と優先順位

本ADRは次を置換する。

- ADR-0005 / ADR-0006の「🌀はCriterionと別のCandidate単位常設単一懸念」という決定。
- Event詳細1画面へ回答者セレクター、判断基準、全Candidateの回答行をまとめる構造。
- 3状態の意味を説明する可視ラベルをCandidateカードへ常設する初期案。

競合時の優先順位は、**ADR-0007 > ADR-0006 > ADR-0003 / ADR-0004 / ADR-0005の置換対象部分 > 旧Slice文書**とする。属性撤廃、Criterion共有リスト、回答者行共同編集、4状態Vote、3状態判定、Candidate作成時刻だけの表示は維持する。

## 影響

- `concerns`のFK、一意制約、index、同一Event guard、RLSテストを新しいCriterion単位へ変更する。
- 完全読取モデルは`hasConcern`ではなく、回答者ごとの`concernCriterionIds`を持つ。
- 画面・E2Eを「オーナー初期セットアップ / ゲスト名前選択 / 候補一覧ダッシュボード / 候補編集」に分けて検証する。
- トップ下部のイベント一覧、広告、マイイベント管理は本スライスで実装しない。
- Supabase Auth、Realtime、service role、local JSON fallback、依存更新は導入しない。
