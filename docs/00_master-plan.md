### 0-2. Pro プランの使用量運用

- Chat / Cowork(Claude Pro) は **同一の使用量プールを消費**する（5時間ごとのセッション上限+週次上限あり）。実装は **Codex（別ツール）** で行う。
- Cowork は Chat より消費が速い。よって:
  - **思考・議論・レビュー → Chat**(消費が軽い)
  - **清書・ファイル生成 → Cowork にまとめて一括依頼**
  - **実装 → Codex で1スライスずつバッチ実行**(だらだらセッションを開かない)
- Claude Pro の上限に当たったら5時間待つ。実装週は Codex を午前・午後の2バッチに区切ると計画しやすい。

## 2. Phase 0: 事業構想(Chat 主体)

### やること

1. **リーンキャンバス作成**(Chat で壁打ち → Cowork で `01_lean-canvas.md` に清書)
   - 課題: 複数人での「決め」が投票だけでは終わらない(拒否権・懸念の扱いがない)
   - 独自価値: ○/−/× + ×解消フローによる「全員が許容できる決定」/ 登録不要
   - 収益: 無料+広告 → 将来オーナー向けプレミアム(広告なし+AI解説)
2. **競合分析**(Chat の Web検索を使用 → `02_competitive.md`)
   - 対象: 調整さん、伝助、LINEの投票/日程調整、Doodle 等
   - 観点: 登録不要か / 拒否権の扱い / 属性別評価の有無 / 収益モデル / 弱点
3. **サービス名候補**: 5〜10案 + ドメイン空き状況の確認方針を添える
4. **判断**: おしげさんが Go / No-Go / ピボットを決定 → 結論と理由を `adr/0001-go-decision.md` に記録

### Chat 用プロンプト雛形

```

題材: 複数人の意思決定支援サービス(仕様概要は添付)。
タスク: リーンキャンバスの9項目を埋める案を2パターン
(広告先行型 / プレミアム先行型)で提示し、各項目に根拠を1行で付けよ。
出力: Markdown表のみ。freeform散文は不要。
```

## 3. Phase 1: 要件定義(Chat で詰め → Cowork で文書化)

現時点で仕様はかなり固まっているため、このPhaseの主目的は **構造化・穴埋め・MVP境界の確定** の3つ。

### 3-1. 要件定義書 `03_requirements.md` の章立て

1. **ユーザー種別**: オーナー(お題作成者)/ ゲスト(URL参加、登録不要)/ ログインユーザー(任意)
2. **機能要件**(それぞれに受け入れ条件を付ける)
   - お題作成: 属性選択(食事/宿泊/アクティビティ/そのた)、共有URL発行
   - 候補管理: タイトル+URLのみ。オーナー・ゲスト双方が追加可
   - 総合評価: ○/−/× の3択、**デフォルトは −**
   - 確定ロジック: 確定行為は扱わず**確定候補を自動ハイライトのみ**。○1つ以上・○最多が対象、×が1つでもあれば除外し**イシュー化を視覚明示**（「理由確認中」等テキスト不使用）。×→−は**URLを知る全員が変更可・履歴なし**。詳細は [adr/0003](adr/0003-evaluation-and-decision-logic.md) / [adr/0004](adr/0004-permission-model.md)
   - ❤️評価: 判断と独立して付与可
   - 価格評価: ❤️の有無のみ(デフォルト無し)
   - 属性別の有効❤️(デフォルト無し):
     - 食事: price / taste(おいしそう) / facility(設備) / place(場所)
     - 宿泊: price / facility / place
     - アクティビティ: price / interest(興味)
     - そのた: interest
     - 🌀(非決定のネガ懸念・単一)は全属性で有効
   - コメント: 候補単位でテキスト投稿可(任意・促さない)
   - 任意ログイン: 未ログインで全機能利用可。ログインすると端末・ブラウザをまたいでイベント一覧を閲覧可(調整さん方式)
3. **非機能要件**: モバイル・デスクトップ同格 / 共有URLを知る人のみアクセス可 / 応答速度 / 同時編集の整合性
4. **MVP境界(スコープ外を明記)**: プレミアムプラン(広告なし・AI解説)、通知、複数お題のグルーピング（※広告実装は MVP 内）
5. **未決事項リスト**: 例)×→−変更の履歴表示、候補の削除権限、イベント有効期限、1人が複数票を持てるか 等 → Chat に洗い出させ、おしげさんが1件ずつ決定し ADR 化

### 3-2. データモデル `04_data-model.md`

Chat にドラフトさせる(Event / Candidate / Participant / Vote / HeartTag / Comment / User)。
匿名参加者の識別方式(Cookie/ローカルストレージのゲストトークン)を必ず含めること。

### 3-3. 画面一覧

お題作成 / イベント詳細(候補一覧+評価UI) / 候補追加 / 確定状態表示 / (ログイン時)マイイベント一覧。ワイヤーは Chat のビジュアル生成で十分。

### Cowork 用プロンプト雛形

```
docs/03_requirements.md を新規作成せよ。
入力: このタスクに添付する仕様メモと Chat での決定事項。
制約: 全機能に「受け入れ条件」を Given/When/Then 形式で付与。
散文禁止、表と箇条書きのみ。スコープ外セクションを必ず設ける。
未決事項は本文に書かず docs/adr/open-questions.md に分離せよ。
```

## 4. Phase 2: DoD と QAフロー(Cowork)

### 4-1. DoD `05_dod.md` — 2階層で定義

**スライスDoD(機能単位・全スライス共通)**
- [ ] 受け入れ条件(Given/When/Then)を全て満たす
- [ ] スマホ実機幅(375px)で表示崩れなし
- [ ] 主要操作のE2Eテストが自動化されグリーン
- [ ] エラー時にユーザー向けメッセージが表示される(白画面禁止)
- [ ] CLAUDE.md / 要件定義書との差分ゼロ(実装が仕様を勝手に変えていない)

**リリースDoD(MVP全体)**
- [ ] 全スライスDoD達成
- [ ] QAシナリオ(4-2)全件合格
- [ ] 本番環境でオーナー1名+ゲスト2名の実データテスト完了(家族で実施可)
- [ ] 利用規約・プライバシーポリシー掲出
- [ ] 計測(アクセス解析)が動作

### 4-2. QAフロー `06_qa-flow.md`

1. **実装完了ごと**: Codex 自身にユニット+E2Eを実行させる(スライスDoDの一部)
2. **スライス受け入れ**: おしげさんが受け入れシナリオを手動実施。代表シナリオ:
   - S1: オーナーがお題(食事)作成 → URL共有 → ゲストが未登録で参加できる
   - S2: ゲストが候補追加 → 全員の初期評価が − になっている
   - S3: ○最多候補が自動ハイライトされる(確定行為なし)
   - S4: ×が1つある候補は○最多でもハイライトから除外され、イシュー化が視覚明示される(「保留」等テキストは不使用) → コメントで理由確認 → ×→−に変更 → ハイライトに戻る
   - S5: ×→−の変更は URLを知る全員が可能で、変更するとハイライトに戻る
   - S6: 価格・属性別❤️がデフォルト無しで、付け外しが即時反映される
   - S7: ログインすると別端末でも自分のイベント一覧が見える
3. **回帰チェック**: スライス追加のたびに S1〜前スライスまでのE2Eを自動再実行
4. **バグ処理**: Blocker(確定ロジック・データ消失系)は即修正、Should は backlog 化しおしげさんが優先度判断

## 5. Phase 3: 開発(Codex)

### 5-1. 技術スタック決定(Chat → ADR)

**決定済み（[adr/0002-tech-stack.md](adr/0002-tech-stack.md)）**: **Next.js + Supabase(Postgres/Realtime、Authは不使用) + Vercel Pro**、ドメインは `kimenosuke.com`。トークン識別で認証不要（ログインMVP外）、Supabase Realtimeでリアルタイム同期、Vercel Proで広告の商用利用OK。

### 5-2. CLAUDE.md(Cowork で作成・保守)

含める内容: サービス一言説明 / 確定ロジックの正確な仕様(○/−/×と×解消フロー) / スタックと規約 / docs/ の参照先 / 「仕様変更を勝手にしない。矛盾を見つけたら実装せず質問すること」という指示。

### 5-3. 垂直スライス計画(1スライス = Codex の1バッチ)

| # | スライス | 含む機能 | 受け入れの軸 |
|---|---|---|---|
| 1 | お題作成と共有 | 属性選択、URL発行、ゲスト閲覧 | S1 |
| 2 | 候補管理 | 候補追加(オーナー/ゲスト)、タイトル+URL表示 | S2前半 |
| 3 | 総合評価 | ○/−/×、デフォルト−、集計表示 | S2後半・S3 |
| 4 | 確定ロジック | ×でイシュー化(確定候補から除外)、×→−変更(誰でも可)、自動ハイライト | S4・S5 |
| 5 | ❤️とコメント | 価格❤️、属性別❤️、テキストコメント | S6 |
| 6 | 任意ログイン | 認証、マイイベント一覧、ゲストトークン引き継ぎ | S7 |

### 5-4. 1スライスの回し方(定型サイクル)

1. Chat: スライス指示書を生成(該当要件+受け入れ条件+DoD を1ファイルに凝縮)
2. Codex: 指示書を渡して実装+テスト実行(プロジェクトフォルダで起動)
3. おしげさん: 受け入れシナリオを手動確認 → 合格なら次へ、不合格なら差分だけ Codex に戻す
4. Cowork: 仕様との差分・決定事項があれば docs/ と ADR を更新

### Codex 用プロンプト雛形

```
CLAUDE.md と docs/03_requirements.md を読んでから着手せよ。
今回のスコープ: スライス4(確定ロジック)のみ。他スライスに触れるな。
受け入れ条件: docs/06_qa-flow.md の S4・S5。
完了時: E2Eテストを追加・実行し、結果と変更ファイル一覧を報告。
仕様に曖昧さがあれば実装せず、質問リストを出して停止せよ。
```
