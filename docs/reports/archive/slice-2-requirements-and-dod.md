# A. 要件定義

## A-1. 目的とスコープ

**目的**: イベント（お題）に対して、参加者が**候補を追加・表示・編集・削除**でき、**各候補の提案者が分かる**ようにする。

> 提案者を可視化する理由: 意思決定では「**誰がその候補を出したか**」が決定プロセス上の重要ファクター。

**In（Slice 2）**: 候補の追加／一覧表示／編集／削除（2重確認）／**提案者の表示・編集**、候補追加に伴うゲスト参加（Participant 生成・お名前は任意）。

**Out（作らない）**: 総合評価 ○/−/×（Slice 3）、確定ロジック（Slice 4）、❤️/🌀・コメント（Slice 5）、広告、マイイベント一覧、ログイン、**URLからの名称自動導出**（リリース後）。**評価操作UI（○/−/×ボタン等）は一切作らない。**

## A-2. ユーザーと権限（[ADR-0004](../../adr/0004-permission-model.md)・性善説）

| 操作 | 権限 | 参加(お名前入力)の要否 |
|---|---|---|
| 候補の閲覧 | 共有URL（`share_token`）または オーナー編集URL（`owner_token`）を知る人 | 不要 |
| 候補の**追加** | 同上 | **不要**（お名前は任意。提案者は自動紐づけ・後から編集可） |
| 候補の**編集**（タイトル/URL/提案者） | 共有URLを知る全員（**B案**） | 不要 |
| 候補の**削除** | 共有URLを知る全員（**B案**）＋2重確認 | 不要 |

## A-3. 機能要件（受け入れ条件 Given/When/Then）

- **AC-2.1 候補追加**
  Given: 共有URL/オーナー編集URLでイベントを開いている人
  When: **タイトル または URL の少なくとも一方**（両方も可）を入れて「追加」
  Then: 候補が作成される。**お名前入力は強制しない**。追加者のブラウザ（`guest_token`）の Participant を（無ければ）自動生成し、**提案者（`created_by`）をその Participant に自動設定**。お名前は**候補追加フォームの任意欄**として入力でき（空でも可）。**同じブラウザ（`guest_token`）が次にお名前を入れて追加すると、自分の表示名（`display_name`）を上書き更新**する（＝実質の「後から変更」・専用の名前編集UIは作らない／A案）。名前を変えると、**自分が出した既存候補の提案者名も一括で変わる**（同一 Participant のため）。**追加に確認ダイアログは挟まない**（新規作成のため）。

- **AC-2.2 候補表示**
  Given: 候補が1件以上存在 / When: イベント詳細を表示
  Then: 各候補は**タイトルがあればタイトル、無ければURL**を表示（URL があればリンク）。**提案者（お名前）を各候補に表示**し、**未設定は「ー」で表示**する。

- **AC-2.4 候補編集（タイトル / URL）**
  Given: 共有URLを知る人（B案） / When: タイトルまたはURLを変更して確定
  Then: **要素ごとに**「**変更します、よろしいですか？**」の確認を挟んでから確定（誤操作・混乱防止）。編集後も「タイトル or URL のどちらか一方は必須」を満たす。保存は last-write-wins。

- **AC-2.6 提案者の編集**
  Given: 共有URLを知る人（性善説） / When: 提案者を**プルダウンで参加者から選び直して**変更
  Then: 変更確定前に「**変更します、よろしいですか？**」の確認を挟む。記入は強制しない。**プルダウンで選べるのは「既存の参加者」＋「ー（未設定に戻す）」のみ**（新しい名前の追加はしない・MVP割り切り）。

  > **AC-2.4／2.6 共通**: 既存要素（タイトル・URL・提案者）の**変更は要素ごとに**「変更します、よろしいですか？」で確認してから確定する。
  > さらに、**Slice 1 のオーナーによるイベント名・メモの編集にも同じ確認を適用**し、編集操作の挙動を全体で統一する（`EventView` に手を入れる。漢字優先化改訂と同じバッチで対応）。

- **AC-2.5 候補削除**
  Given: 共有URLを知る人（B案） / When: 候補を削除
  Then: **2段階の確認**を経て**物理削除＋カスケード**。**1回目**「この候補を消しますか？」→「消す」／「キャンセル」。**2回目**「本当によろしいですか？」→「消す」／「キャンセル」。2回目は**より強い警告色**にし、文言・配色の両方で2段階と分かるようにする（誤削除防止）。

- **AC-2.3 初期評価−（評価UIは作らない）**
  Given: 候補が追加された / When: 参加者から見る
  Then: 評価はまだ存在しない（初期＝−の状態）。**Slice 2 では評価操作UIを一切置かない**（評価・`votes` テーブルは Slice 3）。「強制・催促しない」は上位原則として Slice 3 にも適用。
  - **Slice 2 の検証**: `votes` テーブルは未作成のため「vote行なし」を照会できない。よってDBアサーションは **(a) candidates/participants 行が作られ、提案者(created_by)が設定される** ＋ **(b) 画面に評価操作UI（○/−/×）が存在しない** を確認する。
  - 「**vote行なし＝−**」のDBアサーションは、`votes` が実装される **Slice 3 のQAで行う**（本項の依存を Slice 3 に持ち越す）。
  - **Slice 2 の明示メッセージ**: 「**候補はいつでも追加できます**」（未評価・未参加でも追加可能）を、適切な位置に控えめに表示し、「どれも評価できず行き詰まる」状態を防ぐ。
  - **Slice 3 先取りメモ（Slice 2では実装しない）**: 参加者が評価した後は「**変更や追加評価はいつでもできます**」を常時表示する（Slice 3 で実装）。

## A-4. データ／RLS 要件（新規 migration・既存は編集しない）

### candidates テーブル（[04_data-model](../../04_data-model.md) を改訂）

| 列 | 型・制約 | 変更点 |
|---|---|---|
| id | PK | |
| event_id | FK→events, on delete cascade | |
| title | **任意（NULL可）** | 必須→任意 |
| url | 任意（NULL可） | |
| created_by | FK→participants（提案者・**NULL可**、`ON DELETE SET NULL`） | 作成時は追加者を自動設定。プルダウンで「既存参加者＋ー（＝NULL）」に変更可 |
| created_at | | |

- **制約**: `CHECK (title IS NOT NULL OR url IS NOT NULL)`（少なくとも一方必須）。空文字は NULL 相当に正規化。
- **提案者「ー（未設定）」は `created_by = NULL`** で表す。

### participants（[04_data-model](../../04_data-model.md) を改訂）

- `display_name` を**任意**に（候補追加では入力を強制しない）。後から入力・編集可。

### RLS（Slice 1 のトークンヘッダ方式踏襲）

- **participants ポリシー**:
  - **insert（ゲスト参加）追加**: `guest_token` 一致・当該 event が `share_token` で解決可能。
  - **select 追加**: `share_token`/`owner_token` 保持者に開放（**提案者名の表示・提案者プルダウンに必要**。Slice 1 は自分/オーナーのみだったため拡張）。
- **candidates ポリシー**: select（`share_token`/`owner_token` 保持者）／insert（有効な `guest_token` の Participant が存在し `created_by` 一致）／update・delete（`share_token` を知る全員＝B案。delete は**新規作成**）。
  - **提案者付け替え（`created_by` 更新）の検証**: 更新先 `participant_id` が **NULL（未設定）** または **対象 `event_id` に属する Participant** であることを RLS/トリガーで保証（他イベントの participant を指定できないようにする）。
- **既存 migration は編集しない**。新規 migration で追加。

## A-5. UI 要件（確定文言・漢字優先化・[ui-copy-decisions.md](../ui-copy-decisions.md)）

| 項目 | 確定文言 |
|---|---|
| 候補追加 見出し | 候補を追加 |
| タイトル placeholder | **「例）候補の名前 など」**（汎用・確定）。※属性撤廃（[ADR-0005](../../adr/0005-drop-attribute-dynamic-criteria.md)） |
| URL placeholder | リンク |
| 追加ボタン | 追加 |
| お名前入力（任意） | ラベル「お名前」（全画面共通・任意） |
| 提案者 表示 | 「提案: {お名前}」／**未設定は「ー」** |
| 提案者 編集 | 参加者から選ぶ**プルダウン** |
| 要素変更の確認 | 変更します、よろしいですか？ |
| 削除 1回目 本文 | この候補を消しますか？（→「消す」／「キャンセル」） |
| 削除 2回目 本文 | 本当によろしいですか？（→「消す」／「キャンセル」・**より強い警告色**） |

- **タイトル・URLはどちらか一方でも可**。両方空は不可のバリデーション。この「一方でOK」であることを、**オーナー・参加者（Participantに限らず未評価/未参加の人も含む）に、適切な位置・メッセージで明示**する（例: 候補追加まわりに「タイトルかリンクのどちらかでOK。候補はいつでも追加できます」等）。
- 削除の2段階確認は**1回目と2回目で配色を変える**（視覚的に2段階と分かる）。
- 「候補」は漢字で統一。オーナー/ゲストは同一レイアウト。

### 候補タイトルの placeholder（汎用・属性撤廃）

> **属性は撤廃（[ADR-0005](../../adr/0005-drop-attribute-dynamic-criteria.md)）。** 属性連動placeholderは廃止し、**汎用の単一文言「例）候補の名前 など」**にする。旧・属性別candidate placeholder（バーベキュー/〇〇レストラン等）は不使用。
- 実装メモ: `event.attribute` は存在しない。属性別placeholderマップは作らず、汎用文言をそのまま使う。

## A-6. 非機能要件

- モバイル（375px）／デスクトップ同格。同時編集は last-write-wins。エラー時ユーザー向けメッセージ（白画面禁止）。認証不使用（トークン＋RLS）。local JSON fallback を作らない。依存はバージョン固定。

## A-7. テストデータ方針（OPEN-1=A案）

- E2E データに識別マーカー（例: タイトル/お名前に `[E2E]` 等）を付け、SQL でマーカー一致を一括削除。認証導入スライスで B案（別 Supabase プロジェクト）へ移行。
