## 4. Gitフローの提案（重すぎず・軽すぎず）

現状は `main` 単線運用である（`dac0f11` の確定でも直接 `main` へコミット・pushした）。個人＋AI実装（Codex）の規模に合う軽量フローとして、次を提案する。

- **ブランチ**: スライス単位で `feat/<slice>` を切り、`main` へは squash / まとめコミットで戻す。長期並行ブランチは作らない。
- **コミット粒度**: 1コミット＝1つの意味単位（用語統一、サマリー追加、など）。docs-only変更とsrc変更はできるだけ分ける（`db-*-status` レポートが採用している「docs-only同期」の考え方を踏襲）。
- **コミットメッセージ**: 既存の `feat:` / `fix:` / `docs:` / `chore:` プレフィックス運用を継続する。
- **マージ前の必須チェック**: `npm run check`（tsc）・`npm run build`・`npm run test:e2e:local`・`git diff --check` の4点をグリーンにしてから `main` へ。
- **本番反映**: `main` への反映を起点にVercelが自動デプロイする運用を基本とし、DBを伴う変更だけは後述の承認ゲートを通す。

過剰なレビュー体制やGitFlowのようなリリースブランチは、この規模では重すぎるため採用しない。「スライス単位のブランチ＋4点チェック＋main反映で自動デプロイ」が過不足のない落としどころである。

## 5. ローカル開発・Vercel本番デプロイ運用

### 5-1. ローカル開発（ADR-0008で基盤整備済み）

Supabase CLIは `2.109.1` に固定され、Docker local stackは `127.0.0.1` bind強制で起動する。正式コマンドは `dev:local` / `dev:remote` と `test:e2e:local` / `test:e2e:remote`。DB操作は `supabase:migration:*` / `supabase:db:*` などのnpm wrapper経由に限定され、生の `supabase db reset` や raw `docker exec` は使わない。この基盤自体は完成しているため、残るのは **手順を人が迷わず再実行できるよう明文化すること**である（姉妹編レポートで、正本のどこに何を書くかを整理する）。

### 5-2. Vercel本番デプロイの明確化ポイント

コード側はVercel Proの自動デプロイで完結する一方、**DB migrationはCLIの自動適用ではなく、SQL EditorでのSQL人間実行**という運用になっている（ADR-0008／`db-*-status`）。したがってデプロイ手順は「コード反映（自動）」と「DB反映（承認後に人間がSQL Editorで実行）」を明確に二段で扱う必要がある。次の点を手順として固定することを推奨する。

- コード変更のみのスライスは、4点チェック後に `main` 反映＝自動デプロイで完了とする。
- DBを伴うスライスは、local検証 → remote適用（別承認・人間がSQL Editor実行）→ postflight/Advisor確認 → Production smoke、の順を守る。`supabase db push` / `--linked` / migration history repair は使わない。
- 本番反映後は、Production UIの主要導線（Event作成・Owner編集・Participant・Candidate・共有側権限境界・reload保持）と375×812／1366×768のVisual QAを最小smokeとして実施する。

### 5-3. 残っている運用タスク

remote／ProductionのE2Eデータcleanupは、過去の件数・UUIDを再利用せず **新profileでSELECT-only discoveryから再開** する必要がある。ROLLBACK／COMMITは引き続き別承認とする。cleanup完了をもって現フェーズの区切りとし、latest `origin/main`・Vercel deployment・clean DBの状態を記録する。

## 6. 広告実装の前提（着手は最後段）

広告はMVP境界内の項目だが、現行方針どおり **他のUI/UX・運用整備が済んだ後段** に置く。着手前に次を満たすことを条件とする。

- **スパム・セキュリティ対策**: 登録不要・URL共有という設計上、候補URLや投稿テキストが外部誘導・スパムに使われうる。広告と同居させる前に、投稿内容の扱い（外部URLの免責、noindex維持）と最低限のレート制御・不正利用対策の方針を決める。
- **KPI整理**: 広告の価値判断には計測が要る。アクセス解析（プライバシーポリシーと整合）を先に入れ、イベント作成数・候補数・回答者数・×解消フローの利用率などを、広告前のベースラインとして取得する。
- **プレミアムとの整合**: 将来のオーナー向けプレミアム（広告なし＋AI解説）と両立する形で広告面を設計する。AI解説はAnthropic API（Proとは別契約・従量課金）が必要になる点を予算計画に含める。

まずは無料版で「×解消フローが実際に使われているか」を計測し、そのうえで広告・プレミアムの順序を判断するのが安全である。
